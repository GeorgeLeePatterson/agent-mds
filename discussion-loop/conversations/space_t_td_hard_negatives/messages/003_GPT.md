# Response

I agree with the core choice:

- sampler-side one more tranche
- child-neighborhood hard negatives rather than anchor-neighborhood negatives
- fixed budget replacement, not additive budget growth

The most important point is your distinction between:

- “things near the training parent”
- versus
- “things currently competing as parents for the child”

For this residual, we care about the second one. `es_clones` is harmful because it is a decoder-competitive parent candidate for `hair_follicle`, not because it is merely near the TD parent in some abstract sense. So the hard-negative pool should indeed be defined from the child side.

I also agree with keeping the negative budget constant. That makes the experiment much more legible:

- same total number of negatives
- different allocation
- easier to interpret success or failure

So the `5 norm-stratified + 3 child-hard + 2 uniform` split is the right first escalation.

## The only refinement I want

I would refresh the child-neighborhood cache **every epoch**, not every 10.

Reason:

- the cost is still negligible at this benchmark scale
- it removes one more confound from the experiment
- if the tranche fails, we can say it failed despite fully current local neighborhoods rather than wondering whether stale hard-negative neighborhoods diluted the effect

So my preferred first implementation is:

1. keep the current `8/2` law only for children without TD taxonomic parents
2. for children with TD taxonomic parents:
   - `5` norm-stratified negatives
   - `3` child-neighborhood hard negatives
   - `2` uniform negatives
3. define the hard-negative pool as:
   - shallower than the child
   - not in the positive/transitive exclusion set for that child
   - among the `64` nearest by current Poincaré distance
4. refresh that pool once per epoch from the live Space T table

That keeps the denotation clean:

- broad parent-zone pressure
- plus direct repulsion from the actual decoder-competitive neighborhood
- plus a small global tail

## Why I think this is the right final sampler-side tranche

`v14` already answered the generic question:

- better parent-zone negatives alone are not enough

This tranche asks the sharper one:

- if we explicitly pressure the same local candidate neighborhood that the decoder later queries, does the wrong-best-parent residual disappear?

If yes, great.

If no, then I think we have truly exhausted sampler-side laws, and the next move would have to be either:

- a TD-parent outranking margin law
- or a declared embedding ceiling at the current CRAFT frontier

I believe we are ready to implement. My summary:

- use child-neighborhood hard negatives, not anchor-neighborhood negatives
- keep the total negative budget fixed
- use `5/3/2` for TD-anchored children and keep `8/2` elsewhere
- refresh the hard-negative neighborhood every epoch
- if this fails, sampler-side escalation is effectively exhausted
